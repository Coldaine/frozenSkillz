# Technique: Visual Thesis

Read in Phase 1 for every deliverable. A plan without a written thesis fails Phase 1.

"Adapt style to the subject" is not executable. This is.

## Why this exists

Generic AI styling is the default failure: one house style applied to every subject, indistinguishable from any other model's output. Patrick's visual system is deliberately **polyphonic across projects**; one look across unrelated subjects is a defect, not consistency. Recorded rejections: dark glassmorphism as a system identity, amber/cyan generalized from one project to all, four stat cards as a hero, white landing-page backgrounds by default, a single generic sans carrying everything, every section using one identical card pattern, cosmetic code blocks that add weight without information.

## The workflow

1. **Find real references.** Search for 3+ high-quality artifacts relevant to this subject. Verify they actually look good by rendering or viewing them, not by reputation. Recorded failure: handing over a list of documentation and academic-paper URLs as "visual inspiration" when none of them had good visuals. If you have not seen it, it is not a reference.
2. **Record what they do structurally, not cosmetically.** How do they organize attention? What is their unit of composition? How do they treat evidence? Copying a palette is not using a reference.
3. **Name the metaphor.** One phrase that governs the build. Examples that have worked or been proposed: industrial technical atlas; mission-control console; editorial technical document; auction catalog; laboratory notebook; technical field guide; financial terminal; archival dossier; engineering cutaway; magazine data feature; map-and-legend system. A thesis may combine two or three ("industrial technical atlas + mission-control console + editorial technical document" was the combination that rescued the infrastructure atlas).
4. **Derive the specifics** from the metaphor: typography, hierarchy, the visual object types the deliverable uses, and how evidence is treated. If the metaphor does not change these four things, it is decoration, not a thesis.
5. **Differentiate.** Does this look like the last unrelated deliverable? If yes, the thesis failed; pick again.

## Failing theses

"Clean modern dashboard." "Professional and polished." "Dark mode with accent colors." These name a default, not a direction, and they fail Phase 1.

## Relationship to the five design languages

The pattern-library doctrine (May 25 2026) names five design languages, each intended to have its own library: **editorial, subway, blueprint, terminal, cockpit**. Those languages ARE named visual theses. Once the libraries exist, this step becomes: select a language, which selects that language's pattern library. Until then, the thesis is authored per deliverable using the workflow above.

Polyphony is preserved by language choice. Reuse happens within a language. Pieces operate below the thesis (tables, badges, hover panels, evidence marks); the thesis governs composition, typography, and palette.

## Positive directions worth stealing from

- Editorial comparison tables: alternating row tone, prose-friendly cells, generous but purposeful padding, tables treated as editorial objects rather than GitHub READMEs.
- Margin annotation / sidenote treatment for verification and confidence status.
- Long-form reading signals: paper-toned or tinted backgrounds where the content rewards sustained attention; white signals "landing page."
- Typography that carries meaning rather than one neutral sans doing all the work.
